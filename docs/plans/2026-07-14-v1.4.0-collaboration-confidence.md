# V11 Peer DAW 1.4.0 — Collaboration Confidence

## Release thesis

V11 Peer DAW 1.4.0 should make collaborative editing feel dependable rather
than merely connected.

The release introduces an operation journal for frequent edits, deterministic
incremental application, persisted acknowledgements and retries, and a visible
Sync Center. Full project snapshots remain the bootstrap, compatibility, and
recovery mechanism.

Working title: **Collaboration Confidence**.

## Why this should be 1.4.0

The 1.3.0 release made the application substantially easier to navigate, but
the collaboration model still has one systemic weakness: nearly every local
edit calls `publishProjectChange()`, serializes the complete rig, and can cause
the remote client to rebuild the complete project.

Current characteristics:

- `src/app.js` contains more than 60 project-publication call sites.
- Project Sync protocol 1 supports request, snapshot, update, and acknowledgement
  messages, but not domain operations.
- Concurrent updates converge by whole-project stamp ordering; unrelated edits
  can overwrite one another.
- Acknowledgement state is represented by only the most recent acknowledgement.
- Pending and unresolved work is not visible as a first-class user concept.
- Remote application frequently replaces editor DOM and can interrupt an active
  editing context even when only one scalar parameter changed.
- Arrangement placements are index-addressed and therefore need stable IDs
  before they can be safely edited incrementally.

These concerns affect mixer, clips, arrangement, piano roll, sequencers,
samplers, and module editors simultaneously. Solving them gives more value than
adding depth to one individual module family.

## User-facing outcome

After 1.4.0:

1. Two users can edit different parameters at the same time without one edit
   replacing the other.
2. Frequent edits arrive incrementally without rebuilding the whole rig.
3. The application shows whether work is local, pending delivery,
   acknowledged, conflicted, or recovered from a snapshot.
4. Short disconnects preserve local work and replay it after reconnection.
5. Late joiners still hydrate from a complete snapshot before receiving live
   operations.
6. Legacy or incompatible clients fall back to snapshot synchronization instead
   of silently diverging.

## Proposed UX

### Sync status control

Replace the long project-sync sentence as the only collaboration indicator with
a compact status control in the Session view and transport area.

Suggested states:

- `SYNCED`
- `2 PENDING`
- `RECONNECTING`
- `1 CONFLICT`
- `LOCAL ONLY`
- `RECOVERED`

The current detailed transport summary remains available inside the Sync
Center rather than occupying permanent primary UI space.

### Sync Center

Selecting the status control opens a focused drawer or modal containing:

#### Overview

- Current room and local actor name.
- Snapshot revision and local operation sequence.
- Connected transports and peer counts.
- Pending, acknowledged, retried, rejected, and conflicted operation counts.
- Time of the last remote edit and last successful checkpoint.

#### Activity

A bounded, human-readable timeline such as:

- `Mara changed Main Synth cutoff to 2.4 kHz`
- `You moved Verse Pad to beat 16`
- `2 piano-roll notes replayed after reconnect`
- `Snapshot recovery completed from peer alpha`

This is an operation summary, not the raw packet monitor.

#### Conflicts

Only conflicts that cannot be safely merged should demand attention. Each item
shows:

- The affected object and property.
- Local and remote values.
- Actor and time.
- `Keep local`, `Accept remote`, and `Recover snapshot` actions.

Scalar last-writer-wins changes should normally converge silently, but a recent
remote override of a still-pending local edit should be shown as a low-severity
activity item.

#### Recovery

- Retry pending operations.
- Request a fresh room snapshot.
- Export the local recovery journal.
- Clear acknowledged history.

### Editing feedback

- Do not show a toast for every remote operation.
- Use subtle view-local activity markers for remote changes.
- Preserve focus, selection, scroll, and open editor state when an operation can
  be applied incrementally.
- Reserve banners and warning-colored status for unresolved conflicts or
  prolonged delivery failure.

## Technical design

### 1. Operation envelope

Add a versioned operation message alongside snapshots:

```js
{
  protocol: 2,
  type: 'project-operation',
  messageId: 'alpha:mabc:41',
  clientId: 'alpha',
  sessionCode: 'ROOM-1',
  at: 1784064000000,
  operation: {
    opId: 'alpha:41',
    actorId: 'alpha',
    sequence: 41,
    lamport: 93,
    baseRevision: 18,
    domain: 'module-parameter',
    action: 'set',
    target: {
      moduleId: 'main-synth',
      parameter: 'cutoff'
    },
    payload: {
      value: 2400
    }
  }
}
```

Required properties:

- Stable operation ID.
- Stable target IDs; never transient array indexes.
- Actor-local monotonic sequence.
- Lamport timestamp for deterministic scalar resolution.
- Base snapshot revision for recovery decisions.
- Domain/action pair validated by an operation registry.
- Serializable payload with explicit size limits.

### 2. Domain operation registry

Introduce pure validation and reducer functions rather than applying remote
changes directly inside the application controller.

Proposed modules:

- `src/core/project-operations.js`
  - operation schema and factories
  - target normalization
  - operation validation
  - human-readable summaries
- `src/core/project-operation-reducers.js`
  - pure domain reducers
  - conflict/precondition results
- `src/core/operation-journal.js`
  - outbox, acknowledgement sets, retry metadata, compaction
- `src/core/collaboration-engine.js`
  - protocol negotiation, dispatch, snapshot fallback, replay, diagnostics
- `src/ui/sync-center.js`
  - Sync Center rendering and actions

The application controller should call a small collaboration API and receive
domain events. It should not own protocol sequencing, retry timers, journal
storage, or conflict bookkeeping.

### 3. Initial operation domains

#### Tier A: scalar and bounded-field operations

Implement first because they are frequent and naturally deterministic:

- Module parameter set.
- Effect parameter set.
- Mixer master/channel level, pan, mute, and solo.
- Clock BPM.
- Clip-slot launch/stop and quantization.
- Sampler metadata and bounded cue updates.

Resolution rule: last writer by `(lamport, actorId)` per stable target field.

#### Tier B: stable musical entities

Add stable IDs and operation reducers for:

- Piano-roll note add/update/delete.
- Sequencer/OCRA step set and clear.
- Arrangement placement add/move/resize/delete.
- Arrangement loop-range update.
- Multisampler zone add/update/delete.

Rules:

- Entity operations address IDs rather than indexes.
- Delete uses a bounded tombstone journal during the active session.
- Batch grid edits carry a `batchId` and apply atomically.
- Duplicate delivery is a no-op.

#### Snapshot-only in 1.4.0

The following remain full snapshot changes in this release:

- Whole-project import or example loading.
- Module add/remove.
- Graph and route topology edits.
- Sample binary assignment and library import.
- Complex preset replacement.
- Recovery after an operation precondition failure.

This boundary keeps the release achievable while eliminating the most common
whole-rig rebuilds.

### 4. Stable identity migration

Before Tier B operations can ship:

- Arrangement placements gain persistent `placementId` values.
- Clip slots retain stable IDs through hydration.
- Notes, sampler cues, zones, steps, and automation entities are normalized to
  stable IDs at import time.
- Legacy snapshots without IDs receive deterministic or actor-generated IDs
  once, then serialize them on the next save.
- Reducers reject index-only remote operations.

The project file schema should remain readable by 1.3.0. New IDs are additive;
ephemeral acknowledgement and retry metadata must not pollute exported project
files.

### 5. Operation journal

The journal contains:

- Pending local operations.
- Acknowledgements by connected peer/transport.
- Retry count, next retry time, and last error.
- Recent applied remote operation IDs.
- Bounded tombstones and conflict records.
- Last checkpoint snapshot revision.

Persistence:

- Store per room and actor.
- Prefer IndexedDB through a small storage adapter.
- Keep an in-memory implementation for tests and unsupported browsers.
- Cap retained history by both count and byte size.
- Compact acknowledged operations after a successful snapshot checkpoint.

Suggested defaults:

- Retry with bounded exponential backoff: 0.8 s, 1.6 s, 3.2 s, 6.4 s,
  then degraded status.
- Keep the newest 1,000 operation summaries.
- Compact at 250 acknowledged operations or 2 MiB, whichever comes first.
- Never discard an unacknowledged operation without explicit user recovery.

### 6. Acknowledgement semantics

Each operation acknowledgement includes:

- `ackFor` operation/message ID.
- Applying peer ID.
- Result: `applied`, `duplicate`, `rejected`, or `needs-snapshot`.
- Receiver snapshot revision and Lamport value.
- Optional structured rejection reason.

An operation leaves `pending` when at least one eligible live transport confirms
delivery and all currently known compatible peers either acknowledge it or age
out. Late joiners receive a snapshot checkpoint rather than the entire old
operation journal.

The Sync Center distinguishes:

- Sent but unacknowledged.
- Acknowledged by some peers.
- Fully acknowledged for the current peer set.
- Retrying.
- Rejected and awaiting recovery.

### 7. Snapshot checkpoints and recovery

Snapshots remain authoritative checkpoints:

- Bootstrap and late join always begin with a snapshot.
- The snapshot contains a revision and vector of actor sequences already
  included in the checkpoint.
- Operations older than the checkpoint vector are duplicates.
- A peer requests a fresh snapshot when it cannot satisfy an operation
  precondition or detects a journal gap.
- Applying a recovery snapshot must preserve local unacknowledged operations;
  they are rebased and replayed when valid.

Recovery must be explicit in diagnostics and activity history.

### 8. Protocol compatibility

Protocol negotiation is required because 1.3.0 clients understand only whole
project messages.

Plan:

- Presence and snapshot requests advertise capabilities such as
  `project-ops-v1`.
- Protocol 2 clients continue accepting protocol 1 request/snapshot/update/ack
  messages during the 1.4.x line.
- If any connected peer lacks operation support, the sender may periodically
  provide compatible snapshots while operation-capable peers continue using
  operations.
- Unknown operation domains return `needs-snapshot`; they do not partially
  mutate project state.
- The Session view identifies rooms operating in mixed compatibility mode.

## Architecture boundary

1.4.0 should reduce orchestration pressure in `src/app.js` rather than adding
another collaboration subsystem directly to it.

Target responsibilities:

### `V11PeerDAW`

- Translate UI intent into domain commands.
- Render state and preserve editor context.
- Apply domain events to live audio/module instances.
- Trigger snapshot-only structural actions.

### `CollaborationEngine`

- Create, validate, send, receive, and acknowledge operations.
- Maintain Lamport and actor sequence clocks.
- Coordinate local and Peernet transports.
- Negotiate compatibility.
- Request and apply checkpoints.
- Expose diagnostics as plain state.

### `OperationJournal`

- Persist outbox and acknowledgement state.
- Deduplicate operations.
- Schedule retries.
- Compact checkpoints and bounded history.

### Domain reducers

- Purely transform serializable project state.
- Return changed paths, conflict information, and required live-runtime effects.
- Never access DOM, PeerJS, timers, or audio nodes.

## Delivery slices

### Slice 0 — identity and protocol foundations

- Add stable IDs to arrangement placements and remaining index-addressed
  musical entities.
- Add operation schemas, validation, clocks, and pure journal storage.
- Add protocol capability negotiation without changing live edit behavior.
- Add legacy snapshot migration tests.

### Slice 1 — scalar incremental editing

- Route mixer, module/effect parameter, BPM, and clip launch/stop changes through
  the operation engine.
- Apply remote scalar operations without `rebuildRigFromProject()`.
- Preserve focused control, selection, and workspace scroll.
- Keep snapshot publication for unsupported peers.

### Slice 2 — musical collection operations

- Add note, step, arrangement placement, loop, cue, and zone reducers.
- Add atomic batch operations for grid gestures.
- Add deterministic simultaneous-add/update/delete tests.

### Slice 3 — persisted delivery and recovery

- Persist the per-room operation journal.
- Add acknowledgements, retry/backoff, reconnect replay, checkpoint vectors,
  and snapshot recovery.
- Ensure local pending work survives reload and reconnect.

### Slice 4 — Sync Center UX

- Add compact global sync status.
- Add Overview, Activity, Conflicts, and Recovery sections.
- Add conflict-resolution and manual-recovery actions.
- Replace the permanent verbose sync sentence with an expandable diagnostic
  presentation.

### Slice 5 — compatibility and hardening

- Test mixed protocol rooms.
- Add high-volume operation convergence and journal-compaction tests.
- Add hub loss/re-election, duplicate transport delivery, and dropped
  acknowledgement tests.
- Document operation domains and compatibility behavior.

## Acceptance criteria

### Correctness

- Two peers applying 100 interleaved operations converge to byte-equivalent
  serializable project state.
- Concurrent edits to different fields both survive.
- Concurrent edits to the same scalar field resolve identically on every peer.
- Duplicate local/Peernet delivery applies an operation exactly once.
- Batch note/step operations are never partially applied.
- Late join hydrates from a checkpoint and then accepts subsequent operations.
- A missing operation gap triggers snapshot recovery rather than silent drift.

### Reliability

- Pending local operations survive reload and reconnect.
- Dropped acknowledgements cause bounded retries without duplicate mutation.
- Journal compaction never removes pending operations.
- Legacy project snapshots import and receive stable IDs.
- Protocol 1 clients continue receiving compatible snapshots.

### UX

- Sync state is understandable without reading the packet monitor.
- Pending, retrying, conflict, local-only, and recovered states are visible.
- Users can request recovery and resolve non-mergeable conflicts.
- Incremental remote parameter edits do not replace the focused editor DOM.
- Remote changes do not steal focus or reset workspace scroll.

### Performance

- Scalar operation payloads are at least 90% smaller than the equivalent full
  project update on the default rig.
- A remote scalar operation causes zero whole-rig rebuilds.
- Applying 100 queued scalar operations remains responsive on the supported
  low-power profile.
- The operation journal remains bounded after checkpoint compaction.

### Architecture

- Protocol and journal timers do not live in `src/app.js`.
- Every operation domain has pure reducer tests.
- Collaboration diagnostics can be tested without a browser or PeerJS.
- `src/app.js` is reduced meaningfully rather than expanded by the release.

## Test matrix

### Unit

- Operation schema validation and payload limits.
- Lamport ordering and actor-sequence gaps.
- Idempotency and duplicate delivery.
- Scalar last-writer resolution.
- Stable-entity add/update/delete and tombstone behavior.
- Atomic batch application.
- Journal persistence, retry scheduling, and compaction.
- Checkpoint-vector replay and rebase.
- Protocol capability negotiation and downgrade.
- Human-readable activity summaries.

### Browser integration

- Two local BroadcastChannel clients edit distinct parameters simultaneously.
- Two fake PeerJS clients edit the same parameter and converge deterministically.
- A disconnected client edits, reloads, reconnects, and replays its outbox.
- A dropped acknowledgement retries and applies only once.
- A late joiner receives snapshot plus incremental operation.
- A protocol 1 peer receives fallback snapshots.
- Sync Center reflects pending, acknowledged, retrying, conflict, and recovered
  states.
- Focus, selection, and scroll remain stable while remote operations arrive.

### Soak

- Repeated hub loss and re-election while operations are pending.
- 1,000 mixed scalar and collection operations with duplicate cross-transport
  delivery.
- Repeated room switching verifies journal isolation by room.
- Long session verifies bounded retained operation history and no timer leaks.

## Explicit non-goals for 1.4.0

- Full CRDT semantics for every project object.
- Sample binary transfer reliability and resumable media transfer.
- Real-time shared transport clock or sample-accurate network playback.
- Presence cursors inside piano roll or arrangement.
- Mixer peak/RMS meters, buses, sends, or grouping.
- Module add/remove and route topology as incremental operations.
- Collaborative automation-curve editing.
- Server-authoritative rooms or user permissions.

These remain candidates for 1.5.0 and later.

## Release risks

### Risk: partial operation coverage confuses users

Mitigation: operation-capable and snapshot-only changes share the same visible
journal, and snapshots are explicitly labeled as checkpoints/recovery events.

### Risk: operation replay mutates live audio twice

Mitigation: reducers are idempotent and runtime effects are emitted only when a
new operation changes serialized state.

### Risk: mixed-version rooms diverge

Mitigation: explicit capability negotiation, periodic compatible checkpoints,
and a visible mixed-mode indicator.

### Risk: journal storage grows indefinitely

Mitigation: bounded history, checkpoint vectors, byte/count caps, and tested
compaction that never drops pending work.

### Risk: architecture extraction delays user-visible value

Mitigation: Slice 1 must ship an end-to-end incremental parameter edit before
the broader operation catalog is expanded.

## Suggested release sequence

1. `1.4.0-alpha.1`: stable IDs, operation schemas, journal, scalar local tests.
2. `1.4.0-alpha.2`: two-peer scalar operation path and protocol downgrade.
3. `1.4.0-beta.1`: notes, steps, arrangement operations, persisted replay.
4. `1.4.0-beta.2`: Sync Center and conflict/recovery UX.
5. `1.4.0-rc.1`: compatibility, soak, payload/performance gates.
6. `1.4.0`: documented protocol, clean migration, full release gates.

## Definition of done

1.4.0 is done when collaboration can be inspected, recovered, and trusted:
frequent edits merge incrementally, pending work survives interruption,
conflicts are explicit rather than silently destructive, legacy clients degrade
to snapshots, and users can understand sync health without opening the raw
packet monitor.
